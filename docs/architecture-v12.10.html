Vision & but

Bot multi-stratégies sur Hyperliquid (DEX accessible depuis la France), opérant sur 35 altcoins en perps cross 2× leverage. Combine 5 stratégies orthogonales détectant des inefficiences court-terme (4-72h) sur le marché alt.

3 instances : Paper $1000 Live $641 réel Junior $300 (HL agent wallet). Toutes alimentées par le même code, runtime indépendant.

Principe de design : conviction modérée (~6 positions max), pas de leverage agressif, exits multiples redondants, audit complet en SQLite, alerts Telegram, dashboard temps réel.

1500+ règles testées
5 stratégies survivantes walk-forward
0 retours d'overfit en 13 mois
v12.10.8 after ~250 versions

5 stratégies actives

Toutes ont passé walk-forward strict 4/4 (28m / 12m / 6m / 3m). Ranking par z-score : S9 (8.71) > S8 (6.99) > S1 (6.42) > S5 (3.67) > S10 (3.66).

S1 — BTC momentum → LONG alts

Quand BTC affiche une tendance haussière forte sur 30j (BTC30d > +2000 bps), longue des alts. Macro strategy, slots limités à MAX_MACRO_SLOTS=3.

Modulator v11.10.0 : taille ajustée par 1 + α × btc_z avec α=+0.5 (amplifie en bull, neutre en bear).

Dormant depuis ~12 mois (BTC30d négatif structurellement). Réactivation attendue lors d'un futur bull market.

S5 — Sector divergence (LONG ou SHORT)

Détecte les tokens qui divergent fortement de leur secteur (Infra, DeFi, Meme, AI, etc.) avec un vol_z élevé. Fade la divergence : si le token surperforme le secteur → SHORT ; si sous-performe → LONG.

Adaptive modulator : α=+0.5 sur LONG (bear-favored), α=-0.5 sur SHORT (bear-favored). Tableau directionnel ADAPTIVE_ALPHA_DIR={(S5,-1):-0.5} v12.2.0.

Mécaniques mid-trade :

  • traj_cut v12.7.1 — coupe quand trajectoire désespérée en bear (régime-conditionné)
  • dead_timeout — coupe à T-12h si pinné MAE

Stratégie la plus actuellement active. Bimodalité documentée (winners long-tail vs catastrophes), modulator + traj_cut filtrent les catastrophes.

S8 — Capitulation flush → LONG

Détecte les conditions multi-critères de capitulation : drawdown extrême + vol_z élevé + ret_24h négatif + btc_7d négatif. Achète le bottom des flush events.

Mécaniques mid-trade :

  • s8_inlife_trail v12.5.30 — MFE-conditioned trail, 3 buckets par btc_z (bear : trail tight à 1500/100, neutral : 300/300, bull : 1500/100). Null-shuffle z=+10.52.
  • s8_dead_in_water v12.6.0 — coupe à T+8h si MFE ≤ 50 (signature "rebond mort-né")

Stop loss spécifique : STOP_LOSS_S8=-750 bps (vs STOP_LOSS_BPS=-1250 par défaut).

S9 — Extreme move fade (±20% / 24h)

Quand un token bouge de plus de 20% en 24h, fade ce mouvement (mean reversion sur les overshoots extrêmes).

Modulator α=-0.5 (bear-favored — funding extrême corrèle avec bear regime).

Protections :

  • S9_EARLY_EXIT — coupe à T+8h si current ≤ -500 bps
  • S9_ADAPTIVE_STOP — stop calculé selon volatilité du token
  • runner_extension v11.7.32 — extend hold +12h si MFE ≥ 1200 et current/MFE ≥ 0.3 (capture la continuation)

S10 — Squeeze + faux breakout fade

Pattern technique en 3 étapes :

  1. Squeeze : 3 bougies 4h avec vol_ratio ≤ 0.9
  2. Faux breakout : 4e bougie sort de ≥ 0,5× du range
  3. Réintégration : retour dans la zone de squeeze sous 2 bougies

Fade le breakout (fade-the-fakeout). SHORT-only (LONGs blacklisted : 45% WR sur 28m). Whitelist 13 tokens : AAVE, APT, ARB, BLUR, COMP, CRV, INJ, MINA, OP, PYTH, SEI, SNX, WLD.

Trailing stop v11.4.0 : exit à MFE−150 bps quand MFE > 600 bps.

Modulator macro (v11.10.0+)

Adjusts position size by 1 + α × btc_z where btc_z = rolling 6-month z-score of BTC's 30-day return.

Stratégieα globalα directionnelEffet
S1+0.5—Amplifie en bull (BTC momentum)
S8-0.5—Amplifie en bear (capitulation favor bear)
S9-0.5—Amplifie en bear (extreme moves favor bear)

Mécaniques d'exit

Exits chained en cascade dans trading.check_exits (~10 mécaniques actives), checked toutes les heures sur prix locaux (pas dépendant de HL response).

Ordre des checks (priorité descendante)
  1. catastrophe_stop — -1250 bps (default) / -750 bps (S8). Hard cap, protection ultime.
  2. manual_stop_set — stop manuel dollar via dashboard 🎯 button
  3. s9_adaptive_stop — stop volatility-aware pour S9
  4. s9_early_exit — T+8h, current ≤ -500
  5. runner_extension — extend hold S9 si MFE ≥ 1200 et current/MFE ≥ 0.3
  6. s10_trailing — MFE - 150 bps si MFE ≥ 600
  7. s8_inlife_trail — bucket par btc_z, MFE trail
  8. s8_dead_in_water — T+8h, MFE ≤ 50 (S8 only)
  9. traj_cut — trajectoire désespérée S5 en bear (régime-conditioned)
  10. dead_timeout — T-12h, MFE ≤ 150, MAE ≤ -500, current ≤ MAE+300
  11. timeout — hold expiré (S5 48h, S8 60h, défaut 72h, S9 48h, S10 24h)
  12. mtm_final — fin de backtest window (BT only)

Audit DB events — la query universelle

Depuis v12.10.8, les 8/8 raisons SKIP sont loggées en DB. Pour répondre à "pourquoi le bot n'a pas pris X ?" :

sqlite3 analysis/output_live/reversal_ticks.db "
  SELECT datetime(ts,'unixepoch') as t, symbol,
         json_extract(data,'$.strategy') as strat,
         json_extract(data,'$.reason') as reason
  FROM events
  WHERE event='SKIP' AND ts > strftime('%s','now')-86400
  ORDER BY ts DESC LIMIT 50;
"

Sécurité & wallet model

Dashboard auth

  • Form login + HMAC-signed stateless cookies (30j expiry)
  • Rate limit : 10 tentatives / 5min / IP
  • AUTH_SALT pour empêcher offline replay d'un cookie volé
  • Server-side revocation epoch (v12.5.29) : /logout invalide tous les anciens cookies
  • Telegram alert sur LOGIN OK + failed (sauf interne 127.0.0.1)

HL wallet model

  • Live : HL_PRIVATE_KEY = wallet single-key (signe ET détient les fonds)
  • Junior : API agent wallet model. JUNIOR_HL_PRIVATE_KEY signe, mais HL_ACCOUNT_ADDRESS (master wallet hardcodé dans start_bots.sh) détient les fonds. Junior agent doit être autorisé par master via HL Settings → API (expirations configurables, défaut 6 mois).
  • Paper : 0 SDK dependency, simulations en mémoire

SDK runtime patch (v12.9.6)

HL spotMeta contient parfois des entries cassées (@367 → tokens=[479, ...] hors range). exchange.py:_apply_hyperliquid_sdk_patches wrappe Info.__init__ au boot pour fetcher + sanitizer spot_meta avant délégation au SDK. Idempotent, persisté dans notre code (immune à pip --force-reinstall).

Forces du bot

1. Diversification stratégique orthogonale

5 stratégies aux mécaniques mécaniquement différentes (momentum macro, mean-reversion sector, capitulation flush, extreme fade, squeeze pattern). Survivent au walk-forward indépendamment. Très peu de corrélation entre leurs déclencheurs → portfolio diversifié sans surface commune d'overfit.

2. Discipline anti-overfit rigoureuse

Toute règle shipped a passé : (1) split train/test, (2) Monte Carlo z > 2.0, (3) integration portfolio, (4) walk-forward strict 4/4 sur 28m / 12m / 6m / 3m, (5) null-shuffle confirmation. Beaucoup de candidats rejetés via cette pipeline (cf. docs/bot.md section "Recherche" — 26 backtests, registre anti-reprise).

3. Audit trail complet

Tous les événements significatifs sont loggés en SQLite (OPEN, CLOSE, SKIP avec 8/8 raisons, alerts, supervisor reports). Reproductibilité totale via les snapshots horaires des features et OI. Les R&D futures peuvent rejouer chaque décision.

4. Live ↔ Backtest convergence forte (v12.10+)

Audit btlive sur 65j a quantifié et corrigé les sources de divergence. Le gate 4h (v12.9.0) + boundary scan (v12.10.3) + universe align (v12.9.7) ramènent l'écart attendu à ~$50-100 sur 65j vs $316 historique.

5. Risk management multi-couche

Stops catastrophe ultimes (-12.5%), slot caps (positions/direction/macro/token/sector), cooldown 24h post-close, blacklist tokens net-négatifs, OI gate pour LONGs, dead-timeout pour pinned losers, traj_cut pour S5 catastrophes bear-régime. Aucun chemin unique vers la ruine.

6. Observabilité multi-canal

Dashboard temps réel + Telegram alerts time-critical + Supervisor LLM quotidien + Drift monitor hebdomadaire. Le bot ne peut pas dériver sans qu'on le sache rapidement.

7. Adaptive modulator macro (v11.10.0)

Sizing ajusté par btc_z (régime). Amplifie S1 en bull, S8/S9 en bear. Validé par null-shuffle (z=+10.52 sur S8 in-life). Permet au bot de s'auto-ajuster aux cycles macro sans intervention.

Faiblesses & risques

1. Dépendance Hyperliquid (single point of failure)

Tout repose sur HL : pas de diversification d'exchange. Si HL est down (observé ce matin : SDK timeouts intermittents) ou geofencing change, le bot est paralysé. v12.9.6 SDK patch atténue les incidents data, v12.9.1 timeout bump 60s atténue la slowness, mais le risque structurel demeure.

2. Capital trop bas pour optimum strat (~$15k seuil slippage)

Le BT modélise 4 bps slippage RT. À $15k+ par bot sur tokens thin, le slippage réel dépasse. Memoire project_slippage_ceiling : "Backtest surestime les ROI au-delà de ~$15k capital (slippage alts thin non modélisé)". Live $641, Junior $300 sont en-dessous → BT prédiction fiable. Mais croissance future devra introduire un slippage modeling plus fin.

3. Bimodalité S5 LONG non résolue

S5 winners long-tail vs losers catastrophiques. Multiples tentatives de mid-trade cut (dead_t8h, disp_inlife, cluster split) ont toutes échoué en walk-forward strict. Seul traj_cut (bear-régime only) survit. En bull/choppy, les S5 catastrophes restent non-prévenues. Cf. memoires project_s5_cluster_split, project_s5_disp_inlife_2026_05.

4. Plateau d'optimisation atteint (mai 2026)

3 EDAs négatifs le même jour ont confirmé que la surface "entry-features × thresholds × haircuts" est empiriquement épuisée. Mémoire project_optimization_plateau_2026_05. Pistes futures = nouvelle source de signal (mid-trade features non encore loggées : OI delta, funding flip, BTC delta intra-position) ou nouvelle stratégie S11+. Risque : plus de marge facile d'amélioration.

5. Drift de config entre live et backtests historiques

Le bot évolue rapidement (250+ versions). Les anciens trades live ont été pris sous d'anciennes configs (disp_gate ON, traj_cut absent, etc.). Les BTs régen aujourd'hui ne reflètent PAS exactement ce que le live d'hier a fait. La seule mesure fiable de la performance est le live qui tourne actuellement sous la config courante. Memoire feedback_premise_gate_before_sweep.

6. Path-dependence intrinsèque live vs BT

Même règles, même data, même config : live et BT divergent dès qu'un premier trade a un timing/sizing légèrement différent. Slot remplissage diverge → trade suivant diverge → cascade. Audit btlive : ~$50-100 résiduel sur 65j malgré v12.10+ alignment. Irréductible.

7. Données OI court historique pour 7 tokens v12.7.0

ADA, BCH, DOT, ENA, TON, UNI, XMR : ~2 semaines d'OI history seulement (capturée via live market_snapshots depuis v12.7.0 ship). Sur backtests historiques (12m+), l'OI gate v11.4.9 ne peut pas filtrer les LONGs adverses sur ces tokens → drift de PnL négatif visible. Le live n'est pas affecté (collecte continue), mais comparaison historique faussée.

8. Régime-dependence non encore modélisée explicitement

Quelques règles sont régime-conditionned (modulator α × btc_z, traj_cut bear, S8 in-life par bucket). Mais beaucoup d'autres assument régime stable. Un shift macro brutal (e.g. 2024-Q4 bull → 2025-Q1 crash) peut surprendre les stratégies non-régime-aware. Le supervisor LLM observe, mais ne corrige pas. Memoire project_regime_filter_walkforward_2026_05 : 19 tentatives de regime filter ont toutes échoué walk-forward strict.
