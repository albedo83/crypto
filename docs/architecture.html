Crypto Trading Bot — Architecture v12.6.0

Bot multi-signaux sur Hyperliquid DEX. 28 altcoins, 5 stratégies, paper + live + junior en parallèle.
Version : 12.6.0 Date : 2026-05-15 Modules : 12 + dashboard HTML Stratégies : 5 actives Exits : 10 mécanismes

1. Vue d'ensemble

Bot de trading mean-reversion + momentum sur Hyperliquid (DEX, accessible depuis la France), opérant sur 28 altcoins avec 5 stratégies indépendantes (S1, S5, S8, S9, S10), 2× de levier cross-margin. Trois bots partagent le même code mais avec capitaux/wallets distincts (paper / live / junior) plus un panel admin agrégateur.

Tous les bots tournent le même code Python à partir du même repo. La distinction live/paper se fait par variables d'environnement (HL_MODE, HL_PRIVATE_KEY, HL_CAPITAL, WEB_PORT, HL_OUTPUT_DIR). Permet d'isoler le live de l'expérimentation et de valider tout changement sur paper avant restart live.

6. Sizing & modulator macro

Formule complète

size_usdt = (base_pct × capital) × SIGNAL_MULT[strat] × macro_mult × liquidity_haircut
  where:
    base_pct        = SIZE_PCT (0.18) + bonus si z > 4 (+0.03)
    SIGNAL_MULT     = {S1: 1.125, S5: 3.25, S8: 1.25, S9: 2.00, S10: 2.00}
    macro_mult      = clip(1 + α(strat,dir) × btc_z,
                            MACRO_MULT_MIN=0.3, MACRO_MULT_MAX=2.5)
    liquidity_haircut = {S8: 0.8, autres: 1.0}

  btc_z = z-score 6m du return BTC sur 30 jours
        = (ret_30d_today − mean_180d) / std_180d

α adaptatif par (strat, direction)

Stratα globalα(strat, dir=-1)Effet
S1 (LONG)+0.5—amplifié en bull (×1.5 typique)
S50 (LONG default)SHORT: −0.5S5 SHORT réduit en bull, amplifié en bear
S8 (LONG)−0.5—amplifié en bear (régime favorable à la capitulation buy)
S9−0.5—amplifié en bear pour fades extrêmes
S10——pas de modulator (regime-unstable, v11.10.0)
Le modulator (v11.10.0 + v12.2.0) a été validé walk-forward 4/4 strict + null-shuffle. Coefficients α conservateurs (±0.5 vs ±1.0 optimum) pour limiter overfit. S10 et S5 LONG explicitement exclus car régime-instables.

Floor de sécurité

Si size_usdt < $10 après application du modulator → skip l'entrée (modulator_floor). Empêche les positions ridicules en régime fortement défavorable.

7. Pipeline d'exits (10 mécanismes)

Évalués à chaque scan (~minute). Ordre de priorité dans trading.check_exits :

timeout naturel→ catastrophe stop→ manual stop $→ S9 early exit→ S10 trailing→ S8 dead-in-water→ S8 in-life trail→ dead-timeout

+ manual_close via API et runner_ext S9 qui pousse target_exit de +12h au timeout si winner.

#ReasonCibleTriggerVersion
1catastrophe_stoptousunrealized ≤ stop (−1250 / −750 S8 / adaptive S9)v0
2timeouttousnow ≥ target_exit (72h S1, 48h S5/S9, 60h S8, 24h S10)v0
3dead_timeouttousà T−12h : MFE ≤ 150 ET MAE ≤ −500 ET current ≤ MAE+300v11.7.2 → v12.5.0
4s10_trailingS10MFE ≥ 600 ET unrealized ≤ MFE−150v11.4.0
5s9_early_exitS9hours_held ≥ 8h ET unrealized < −500 bpsexistant
6runner_ext *S9à timeout, MFE ≥ 1200 ET unrealized ≥ 30% MFE → +12h holdv11.7.32
7manual_closetousbouton ✕ dashboard ou POST /api/close/{sym}v0
8manual_stop_settousseuil $ fixé via 🎯 dashboard, net P&L ≤ stop_usdtv12.5.10
9s8_dead_in_water NEWS8 LONGhours_held ≥ 8 ET MFE ≤ 50 bpsv12.6.0
10s8_inlifeS8MFE ≥ activation[regime] ET unrealized ≤ MFE − offset[regime]v12.5.30

* runner_ext n'est pas un exit en lui-même mais une extension du timeout.

S8 dead-in-water vs S8 in-life trail opèrent aux queues opposées de la distribution MFE :
  • Dead-in-water : MFE ≤ 50 bps à T+8h (la position n'a jamais respiré) → cut early
  • In-life trail : MFE ≥ 300 (neutral) ou ≥ 1500 (bear/bull) puis retrace → lock les gains
Distributions strictement disjointes, aucun chevauchement possible. pos.mfe_bps étant monotone non-décroissant, le dead-in-water est naturellement idempotent (une fois MFE > 50, la rule ne fire plus).

10. Auth & sécurité

Niveaux de protection

  1. Hard limits exchange-side : 2× cross margin, séparation spot/perps
  2. Bot-side hard stops : catastrophe stop −1250 bps, max 6 positions, max 4 même direction, max 2 par secteur, blacklist statique
  3. Risk modulators : modulator macro, dispersion gate, OI gate LONG, modulator floor $10, slot reservation macro/token
  4. Manual intervention : /api/close, /api/manual_stop, /api/pause, /api/reset (paper only)

Auth dashboard

  • Login HTML form, DASHBOARD_USER / DASHBOARD_PASS dans .env
  • HMAC-signed stateless session cookies (SHA256 + AUTH_SALT), 30j expiry → survivent aux restarts
  • Rate limit : 10 tentatives login / 5 min / IP (exponential backoff)
  • v12.5.29 : revocation epoch côté serveur — /logout invalide les tokens signés avant l'epoch
  • v12.5.29 : rate-limit per-IP sur endpoints mutating (30/min)

Hardening v12.5.29 → v12.5.31

  • Validation avgPx > 0 sur fills HL (sinon entry/exit_price=0 → P&L corrompu)
  • NaN/inf rejetés sur stop_usdt, amount
  • Boot reconcile 2-attempts (1 cache lag ne drop plus un ghost)
  • Timeout cap sur tous les appels SDK HL via _sdk_call (10-20s) + saturation warning
  • JSON body parse errors → 400 (pas 500 stacktrace)

Kill-switches disponibles

12. Backtest infrastructure

Tous les backtests vivent dans backtests/ et utilisent le même moteur backtest_rolling.run_window. Aucun signal/exit n'est shipé sans validation walk-forward 4/4 strict.

Fenêtres walk-forward

  • 28m — fenêtre longue, capture multi-régime, baseline historique
  • 12m — médium-term, sanity OOS
  • 6m — récent semi-out-of-sample
  • 3m — régime courant, le plus susceptible de casser

Critère "strict 4/4"

Un changement doit gagner du P&L sur les 4 fenêtres simultanément ET ne pas dégrader le drawdown moyen de plus de +0.5pp. Si une seule fenêtre fail → rejet.

Validations supplémentaires (selon ship)

  • Null-shuffle : randomiser une dimension (e.g. btc_z), refaire la règle, vérifier z-score > 3
  • IS/OOS sliding : train sur 18m, test sur 6m, glisser × 4 splits
  • Per-window parameter stability : la règle reste-t-elle optimale dans chaque fenêtre ?
  • Bonferroni : pour les EDA multi-tests, p < 0.05/N_tests
  • Bootstrap stability (clusters) : ARI ≥ 0.65 sur 50 resamples

Hooks utilisés

inlife_exit_extra (callable) — snapshot dict par (position, candle), retourne (True, reason) pour forcer exit. Utilisé par backtest_inlife_exit (S8 trail R&D), backtest_s5_dead_t8h, backtest_s8_dead_in_water.

Règle restart (CLAUDE.md)

JAMAIS de restart sans confirmation utilisateur explicite. Bump VERSION + edit + commit librement, mais le restart (fuser -k + start_bots.sh) reste sous contrôle utilisateur. Une autorisation générique "restart les bots" couvre seulement paper + live. Junior nécessite mention explicite ("restart junior", "tous les 3").
